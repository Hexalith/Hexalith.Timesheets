- UX implementation system decision:
  - Use Hexalith FrontComposer module patterns and Blazor Fluent UI components.
  - Prefer existing FrontComposer, Fluent UI, or Blazor components before any custom implementation.
  - Do not use raw HTML, custom CSS, or JavaScript when an equivalent component exists.
  - Treat Fluent UI's professional themes and look-and-feel as the baseline visual system.
  - Use custom CSS only when mandatory for behavior or layout not covered by available components.
- Source-derived product position:
  - Hexalith.Timesheets is the trusted effort ledger of the Hexalith ecosystem, not a generic timer or spreadsheet.
  - The UX must support actor-neutral effort evidence for internal contributors, external parties, and AI agents.
  - Timesheets owns time evidence, approval state, corrections, activity types, billable flags, AI effort metrics, approved-time ledger, and exports.
  - Timesheets must not present itself as payroll, invoicing, rate-card, revenue-recognition, desktop surveillance, or full external-party portal software.
- Source-derived users and journeys:
  - Internal contributor records time against Project or Work, reviews the week/month, and submits a Timesheet Period.
  - Approver/project manager reviews submitted time, approves valid entries, rejects unclear entries with reasons, and sees correction state.
  - External contributor confirms or adjusts a scoped entry through Magic-Link Confirmation without broad tenant access.
  - AI agent records execution evidence including wall-clock time, model/tool runtime, billable effort, and token consumption.
  - Finance/accounting consumer filters and exports approved billable evidence with stable IDs and correction lineage.
- Source-derived core surfaces:
  - Time entry capture from Project or Work context.
  - Contributor timesheet period review and submission.
  - Approver queue and entry/period approval detail.
  - Rejection/correction review path.
  - External Magic-Link Confirmation page with expired/used/invalid no-disclosure states.
  - Activity Type catalog management for tenant and project scopes.
  - Operational reports for Project, Work, Contributor, Activity Type, Billable Flag, Approval State, period, and contributor category.
  - Approved-Time Ledger export surface.
  - AI effort reporting surface separating human/external duration from AI runtime/token metrics.
- Source-derived behavioral constraints:
  - Entry and period approval states must be distinct.
  - Approved entries are locked from direct edit; corrections are additive and auditable.
  - Submission blocks only invalid entries when possible and shows the correction needed.
  - Projection freshness must be visible when read models are stale, rebuilding, degraded, or unavailable.
  - External magic links are single-use, scoped, expiring, and reveal no tenant/project/work/party/time-entry details when invalid.
  - Party display and personal data are hydrated at read time; durable UX copy must avoid implying Timesheets owns Party identity.
  - AI token metrics must be represented as unavailable/unknown when missing, not zero.
- Source-derived concern scan:
  - Accessibility: target WCAG 2.2 AA on internal UI surfaces.
  - Security: tenant isolation, fail-closed authorization, and no data leakage in invalid magic-link states.
  - Auditability: approvals, rejections, corrections, and exports must expose evidence without implying mutable rows.
  - Data minimization: avoid storing or displaying copied Party/Project/Work data as if Timesheets owns it.
  - Content sensitivity: comments may contain sensitive data; policy is unresolved.
  - Internationalization/time: tenant-aware dates, periods, cutoffs, and time zones are unresolved.
  - Reporting density: operational and finance users need filter-heavy, scan-friendly views.
  - Platform/form factor: unresolved.
  - Dark mode: unresolved; Fluent theme inheritance is the baseline.
  - Offline mode: unresolved and not promised by sources.
  - Motion: unresolved; default should follow Fluent interaction norms unless user decides otherwise.
- Open UX questions from accepted sources:
  - What form factor is in scope: web-only internal shell, external magic-link web page, mobile, or multi-surface?
  - What are the product stakes: internal, consumer, regulated, or regulated-adjacent because of billing/audit evidence?
  - Which named protagonist journeys should be primary for the UX handoff?
  - What tenant time-zone and weekly/monthly period policy should the UI use?
  - What approver precedence should the UI communicate when tenant, project, work, and finance roles overlap?
  - Should corrections be presented as superseding entries, offset entries, or both?
  - Who issues magic links, and does any confirmation require secondary identity verification?
  - Who governs tenant and project Activity Types?
  - Is CSV export enough for v1, or does UX need an API/webhook/export contract surface?
  - Are comments allowed to contain customer/private data, and do they need classification or redaction affordances?
- Fast-path assumptions accepted by "do best":
  - Draft spines should proceed without additional elicitation.
  - Internal UI is responsive web in `FrontComposerShell`.
  - External Magic-Link Confirmation is a minimal responsive web page and may be outside full internal shell navigation.
  - Product stakes are regulated-adjacent because approval, correction, export, tenant isolation, and billing evidence require audit-quality behavior.
  - Fluent UI professional theme inheritance is the visual baseline; no custom Timesheets palette or typography.
  - Tenant time zone controls period boundaries until a more specific policy is provided.
  - Self-approval is denied unless policy explicitly allows it.
  - CSV export is sufficient as a v1 UI affordance unless architecture selects API/webhook export at launch.
  - Corrections are presented as superseding lineage by default; offset entries remain an architecture option.
  - Key-screen HTML mockups are not generated in this pass because the user specifically constrained UX toward FrontComposer/Fluent UI components and away from raw HTML/CSS. Surfaces are specified as spine-only unless later requested.
