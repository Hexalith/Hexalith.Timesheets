# Decision Log: Hexalith.Timesheets Product Brief

## 2026-06-18

- Created a new product brief workspace for `Hexalith.Timesheets`.
- Detected intent as `create` from the user request: `$bmad-product-brief a timesheet module for Hexalith`.
- Loaded BMAD workflow customization from `bmad-product-brief`; no activation prepend or append steps were configured.
- Loaded persistent project context files from sibling Hexalith modules as foundational constraints.
- Established working assumptions for discovery only: Timesheets is likely a Hexalith domain module, should preserve tenant isolation, and should use Hexalith.EventStore for domain persistence.
- Noted `Hexalith.Works` as a nearby product precedent for work lifecycle and burn-down concepts, without treating it as a dependency requirement.
- User stated the module should record time spent on projects or work items.
- User chose the fast path.
- User confirmed Timesheets should support three purposes: personal time capture, billing/accounting evidence, and operational reporting.
- User confirmed internal users, external parties, and AI agents can record time.
- User confirmed a time entry contains date, duration, project/work reference, activity type, comment, billable flag, and approval.
- Market baseline checked: existing time tracking products commonly support project/task time entry, billable/non-billable categorization, approvals, invoicing/export, and reporting. The draft positions Hexalith.Timesheets around event-sourced, actor-neutral time evidence rather than a generic standalone timesheet app.
- User resolved open questions: approval happens both per entry and per weekly/monthly timesheet period; activity types are tenant- and project-scoped; Timesheets owns the billable flag and approved-time ledger but not rates or invoicing; AI-agent entries capture wall-clock execution time, model/tool runtime, billable effort, and token consumption; v1 external-party experience is API-only integration plus magic-link confirmation.
