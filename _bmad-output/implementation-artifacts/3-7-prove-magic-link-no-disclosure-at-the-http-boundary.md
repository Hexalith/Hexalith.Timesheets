# Story 3.7: Prove Magic-Link No-Disclosure at the HTTP Boundary

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a security reviewer,
I want executable HTTP-boundary tests for magic-link invalid states,
so that service-level no-disclosure behavior is proven at the routes external contributors actually hit.

## Acceptance Criteria

1. Given the invalid-state matrix (malformed, unknown, expired, used, revoked, unauthorized, cross-tenant, wrong-recipient, wrong-action, stale-catalog, repeated-token), when the external magic-link routes are exercised **through an in-process HTTP host** — `GET /api/timesheets/magic-links/confirm?t=…`, `POST /api/timesheets/magic-links/confirm/submit?t=…`, `GET /api/timesheets/magic-links/adjust?t=…`, `POST /api/timesheets/magic-links/adjust/submit?t=…` — then every case returns an **equivalent** neutral failure: HTTP **403**, content type **`application/problem+json`**, ProblemDetails `title` = `"This link is expired or unavailable."`, `detail` = `"Request a new confirmation link from the sender."`, equivalent body shape and headers.
2. Given any invalid-state HTTP response from AC1, when its serialized body and headers are inspected, then it reveals **no** tenant, Project, Work, Party, Time Entry, duration, comment, Activity Type, approval state, token existence, expiry reason, or capability state — and contains no token value, derived hash, decoded material, route-specific reason text, `errors` collection, or the unused `RecoveryPath` field.
3. Given a usable token state injected via a test-double loader, when a valid external request is made through the same host, then it produces the **distinguishable** success response (`200 OK` display body / `202 Accepted` submit) — proving the failure-equivalence in AC1 is non-vacuous (the host *can* succeed; all invalid states still collapse to the single 403).
4. Given invalid-link telemetry or diagnostics are emitted, when authorized logs/audit records are inspected, then only correlation ID, timestamp, safe outcome category, and permitted scoped/hash references are present, and token values, decoded material, comments, command bodies, personal data, target names, and raw EventStore envelopes are absent — proven by test and kept green by the diagnostics-privacy fitness lane.
5. Given abuse detection or rate limiting applies (or repeated invalid attempts occur), when a repeated/throttled invalid request is made, then the response remains **externally indistinguishable** from any other invalid-link response (byte-equivalent 403 ProblemDetails, no distinguishing status/headers/body/length) and operational diagnostics remain privacy-safe.
6. Given the change is complete, when the in-scope solution is built and tested, then it compiles clean under `-warnaserror`, all affected test lanes pass, and the existing fitness invariants stay green — exactly one `private static IResult Denied()` and one `Results.Problem(` in the endpoint file, `DiagnosticsPrivacyTests`, and `DependencyDirectionTests` all still pass.

- Do NOT change the `Denied()` helper, denial copy, routes, or command-service logic — assert the existing behavior; keep exactly **one** `Denied()` and **one** `Results.Problem(` in the endpoint file (the fitness test enforces this).
- Do NOT replace/weaken any production DI default; override seams **only** inside the test host's `ConfigureTestServices`.
- Do NOT fabricate a persisted authority or wire the live projection topology to force a happy path — a fail-closed unavailable loader is safer than a fake persisted lookup; drive states through a test-double loader.
- Do NOT let any externally observable channel distinguish failure categories — status, headers, content type, body, body length, or timing must be equivalent across all invalid cases (incl. rate-limited/repeated).
- Do NOT add a 429 or `Retry-After`/`X-RateLimit-*` header for throttling (would leak that throttling occurred); a throttled response must look like any other `Denied()`.
- Do NOT log/persist tokens, derived hashes (where disallowed), decoded material, comments, command bodies, personal data, target names, raw claims, or EventStore envelopes; keep `MagicLinkInvalidLinkOutcomeCategory` Server-internal and out of every external response.
- Do NOT scaffold a UI for NFR13; do NOT add inline package versions; do NOT create `.sln`; do NOT modify sibling submodules; do NOT init nested submodules.
- Do NOT overstate completion: every claimed equivalence must be a passing HTTP-level assertion; report exact test counts and a complete File List.
