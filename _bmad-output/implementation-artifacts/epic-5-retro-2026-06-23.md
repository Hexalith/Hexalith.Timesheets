# Epic 5 Retrospective (Closure): Release Readiness Verification

## Why This Retrospective Exists

Epic 5 is the **final epic of the Timesheets project** and the only one whose mandate is to *verify*, never to
first-implement. The 2026-06-20 readiness repair deliberately rehomed every launch-scope integration out of the old
catch-all Epic 5 into its owning feature story (magic-link loader → 3.6, HTTP no-disclosure → 3.7, Work validation →
1.10, planned-effort → 4.8, export preview → 4.9, command performance → 1.11, report/export/dashboard performance →
4.10). Epic 5 was reframed to a single story — **5.1 Final Launch-Readiness Gate and Documentation Sync** — that
aggregates that evidence, classifies every launch-scope item, syncs the docs to committed reality, and renders a
defensible release verdict.

This retrospective closes Epic 5 and, with it, the Epic 1–5 build. It runs autonomously in YOLO mode; Jerome is
represented by sprint status, the Story 5.1 record, its Senior Developer Review, source code, the docs under `docs/`,
and the planning artifacts. The single most important sentence frames everything below, because it is the lesson the
entire project converged on and the reason Epic 5 exists at all:

> **Story-complete is not launch-ready — and a checked box is not proof.** Epic 5's job was to make that distinction
> durable, mechanical, and honest, so Timesheets cannot ship while quietly claiming integrations the running host
> does not actually perform.

## Key Learnings

1. **A launch claim must match host wiring, not code existence.** A type that exists in `src/` but is not bound in the
   running host is `waived`, not "implemented." `WorksQueryWorkReferenceValidator` is real and the host still denies
   Work writes via `DenyAllWorkReferenceValidator`; both facts are true and the record states both.
2. **Honesty needs a fitness guard, not a good intention.** The project's dominant defect — overstatement — was finally
   answered with a mechanical control: `LaunchReadinessTests` asserts the record carries the classification vocabulary,
   the verdict vocabulary, owner/risk/revisit markers, and a non-vanity verdict, so the gate cannot silently rot.
3. **A presence-guard is necessary but not sufficient.** The fitness test proves the record exists and is shaped
   correctly; it cannot prove the classifications are still semantically true. Flipping a waiver to `implemented`
   requires a human (or review) update — the guard buys structure, not freshness.
4. **`CONCERNS` is a successful outcome for a verification epic.** The defensible verdict for a story-complete system
   with named waivers is `CONCERNS`, not `PASS`. Manufacturing a `PASS` would have been the precise failure Epic 5
   existed to prevent. Letting the evidence set the verdict is the discipline.
5. **Record divergences you are not allowed to fix.** The Dapr/FluentUI version mismatches live in a submodule that is
   out of scope to edit; the right move is a `waived` concern row in `docs/launch-readiness.md`, not silent acceptance
   and not an out-of-scope submodule edit.
6. **Centralizing the waivers is worth the maintenance cost.** Scattered "reserved for later" notes across stories were
   the failure mode the readiness repair found. One fitness-guarded record with owners and revisit conditions converts
   that scatter into a single auditable launch gate — at the cost of one more document to keep current.
7. **Doc-sync belongs in the story that calls out the lag.** Closing the "README/architecture lag the code" pattern in
   the same story that names it is what finally stopped the recurrence. Documentation currency is a deliverable, not a
   footnote.
