# Epic 4 Retrospective: Approved Time Ledger, Reporting & Finance Export

## Epic Summary

Epic 4 delivered the read, reporting, approved-ledger, finance-export, and dashboard foundation for Timesheets.

- 7 of 7 stories completed.
- Operational users can query Time Entries by contributor, target, period/date range, Activity Type, billable state, approval state, correction state, contributor category, and source type.
- Approved-Time Ledger rows are rebuildable from domain events, preserve approval/correction lineage, expose projection freshness, and block export readiness when freshness or disclosure is insufficient.
- Project and Work actual-time reports roll up effort without copying Project hierarchy, Work lifecycle/planning state, Party personal data, or sibling-owned display state.
- AI effort reporting preserves human/external duration, AI wall-clock runtime, model/tool runtime, billable effort, and provider token counts as distinct units.
- Approved billable-time export generates deterministic CSV v1 evidence from disclosed fresh ledger rows and emits safe `ApprovedTimeExported` audit evidence without rates, invoices, payroll, taxes, or revenue-recognition data.
- Dashboard overview composes current-period state, pending actions, approval workload, report shortcuts, ledger readiness, export readiness, and projection statuses over existing query services.

Sprint status showed all Epic 4 story keys as `done`. The Epic 4 aggregate row still said `in-progress` before this retrospective, so this workflow corrected it to `done` and marked `epic-4-retrospective` as `done`.

## Review Pattern Synthesis

The senior reviews found useful issues and turned them into local fixes:

- Story 4.1 established operational query contracts, degraded freshness, list projection behavior, fail-closed list readers, result-level authorization, and metadata. Review noted freshness-mapping duplication and missing `ActivityTypeScope` on query rows for later stories.
- Story 4.2 implemented the Approved-Time Ledger and retired freshness duplication with `ProjectionFreshnessMetadataMapper`. Review fixed export-readiness detail/flag coherence after row filtering and cleaned duplicated sort logic.
- Story 4.3 implemented Project/Work actual-time reports and Works planned-effort adapter boundaries. Review fixed activity-type scope grouping and added missing Works source reference-state metadata.
- Story 4.4 extended report models with AI metrics. Review fixed order-dependent provider token-availability rollup and documented the token-vocabulary architecture guard carve-out.
- Story 4.5 generated deterministic CSV v1 export from disclosed approved ledger rows and metadata-backed export actions.
- Story 4.6 added accepted-export audit evidence, deterministic CSV verification, formula-leading value safety, and explicit documentation that `PreviewApprovedTimeExport` has no dedicated server handler yet.
- Story 4.7 added dashboard overview composition. Review fixed capture shortcut gating, pending-correction count semantics, and brittle unresolved-authority string coupling.

The pattern is healthy: defects were mostly coherence, edge-case, and documentation issues rather than architectural reversals.

## Key Learnings

1. Freshness state must be computed from final disclosed data, not from a pre-filtered projection page.
2. Dashboard write shortcuts must not inherit read-projection freshness gates when the action itself is authority-gated elsewhere.
3. AI evidence needs typed units and availability state; provider token data is not duration, approval evidence, or finance value.
4. Export readiness must combine projection freshness, disclosed row count, billable filter policy, row authorization, and comment/export policy.
5. Fail-closed unavailable readers/providers are acceptable defaults only when docs and tests make their launch-readiness impact clear.
6. Metadata-backed UI descriptors can carry the operational surface, but README and architecture notes must describe what is implemented versus what remains generated/downstream behavior.
7. Story-review findings are valuable future context when they include exact behavioral fixes, not only approval summaries.

## Significant Discovery Assessment

No implementation learning requires rewriting Epic 4's completed scope. The plan was directionally sound, and the implementation stayed inside its module boundaries.

Significant non-blocking discoveries:

- `PreviewApprovedTimeExport` exists as a contract, but readiness currently comes from Approved-Time Ledger query output and export generation results; there is no dedicated server preview handler.
- Works planned-effort reporting is represented through an adapter and unavailable/default state, not a concrete Works integration.
- Dashboard counts compose bounded query results; this is acceptable for current overview behavior but not a substitute for total-count read models if product claims require exact workload totals.
- Performance evidence remains intentionally skipped until realistic EventStore-backed fixtures exist.
- Root README and architecture notes require updates after Epic 4 to avoid understating implemented coverage or overstating integration maturity.

Recommended response:

- Keep Epic 4 marked complete.
- Treat performance, Works adapter maturity, preview-handler decision, and magic-link loader as launch-readiness or future planning items.
- Keep finance/export language anchored in evidence, not invoice/payroll/rate/tax/revenue ownership.
- Keep dashboard behavior read-only and composed over existing disclosure services.

## Commitments

- Keep EventStore as the only authoritative persistence boundary.
- Keep projections rebuildable, idempotent, freshness-aware, and non-authoritative for writes.
- Keep result-level authorization and display hydration ordering intact across list, ledger, report, export, and dashboard surfaces.
- Keep AI units explicit and never convert token/runtime evidence into human hours or finance values.
- Keep finance export as evidence export only.
- Keep unavailable defaults honest in docs, tests, and launch-readiness claims.
