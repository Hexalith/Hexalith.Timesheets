# Epic 2 Retrospective: Submission, Approval, Period Review & Corrections

## Epic Summary

Epic 2 delivered the internal trust workflow that turns captured time into reviewable, approved, rejected, corrected, and period-scoped evidence.

- 8 of 8 stories completed.
- Draft Time Entries can be submitted with EventStore-backed `TimeEntrySubmitted` evidence.
- Approval authority is resolved through a fail-closed policy resolver with source attribution, precedence, same-precedence ambiguity handling, and default self-approval denial.
- Submitted Time Entries can be approved or rejected with durable approver, decision, authority, scope, and reason evidence.
- Rejected entries can be corrected and resubmitted through additive events while preserving rejection history.
- Approved entries are locked from direct mutation and can be corrected only through additive approved-correction evidence.
- Weekly and monthly Timesheet Periods can be submitted, approved, or rejected while keeping period state separate from entry state.
- FrontComposer metadata and OpenAPI schemas were extended without introducing a bespoke Timesheets UI project.
- Final story evidence reported full restore/build success with warnings as errors and passing direct xUnit v3 lanes after local VSTest socket fallback.

Sprint status confirmed all Epic 2 story keys were `done` before this retrospective. The Epic 2 retrospective key was still `optional` before this workflow.

## Review Pattern Synthesis

The senior reviews found and fixed medium-severity gaps before stories were marked done:

- Story 2.1: duplicate entry IDs inside one submission batch could emit duplicate submitted transitions; integration test File List entries were missing.
- Story 2.2: approval authority behavior was sound, but File List omissions recurred and two low follow-ups were recorded around unavailable-provider fall-through and contributor source consistency.
- Story 2.3: approve/reject command services consumed authority resolution and preserved safe denial behavior; review verified self-approval denial and integration coverage.
- Story 2.4: correction authority needed to be explicit and tested as provider-resolved rather than contributor-self-owned.
- Story 2.5: lock bypass paths were hardened so approved and superseded entries cannot be directly mutated through approval/rejection or correction paths.
- Story 2.6: `TimeEntryCorrectionReason` validation was aligned with the published OpenAPI schema; approved correction coverage and File List records were corrected.
- Story 2.7: period boundary calculation was made fail-closed and single-source-of-truth; DST and deterministic entry dispatch tests were added.
- Story 2.8: period approval service tests were expanded for stale entry evidence, cross-tenant entry denial, and self-approval contributor/action forwarding.

The pattern is healthy after correction: reviews found real issues, fixed them, and re-ran the relevant lanes before approval. The remaining risks are policy decisions and runtime evidence, not known broken Epic 2 behavior.

## Key Learnings

1. Period state and entry state must stay separate in both domain decisions and read models.
2. Fail-closed defaults remain useful, but positive paths need explicit configured fixtures and tests.
3. Authority resolution should stay centralized; command services should not grow ad hoc role checks.
4. Additive correction events work for both rejected and approved entries, but policy copy must distinguish correction from editing.
5. Review claims need branch-level proof when stories say stale, unavailable, cross-tenant, duplicate, or self-approval paths are covered.
6. In-process integration tests are valuable workflow proof, but they do not replace runtime EventStore/Dapr/Aspire evidence.
7. Documentation drift shows up quickly after implementation-heavy epics; README, PRD, and boundary docs need verification after each epic.

## Epic 3 Preview

Epic 3 is "External Contributor Confirmation." It lets external contributors submit or confirm scoped time without becoming full internal users, using API-only submission and no-disclosure Magic-Link Confirmation.

Dependencies on Epic 2:

- External API-created entries must enter the same capture, submission, approval, rejection, correction, and locking workflow as internal entries.
- Confirmation must remain contributor evidence, not approval.
- Magic-link issuance and use must reuse fail-closed tenant/resource/Party/Project/Work/Activity Type validation.
- Invalid, expired, used, revoked, unauthorized, malformed, and unknown links must use equivalent no-disclosure behavior.
- External confirmation read models must not leak comments, target names, personal data, raw token values, decoded token material, command bodies, or protected identifiers.

Preparation needs:

- Define the magic-link capability state boundary before implementing issuance.
- Define token hash storage, no raw token logging, and no decoded capability persistence as explicit test cases.
- Build a no-disclosure equivalence matrix before the invalid-link story.
- Decide whether external API submission can create Draft, Submitted, or policy-selected initial states per tenant.
- Keep the first external page minimal and Fluent-compatible without adding internal shell navigation.
- Avoid treating confirmation as approval in command names, copy, events, or read models.

## Significant Discovery Assessment

No implementation learning requires rewriting Epic 3's scope. Epic 3 remains directionally sound.

Significant but non-blocking discoveries:

- Period approval currently fails closed for already-approved included entries unless a future carry-existing-approval policy allows them.
- Required-comment and partial-submission policy vocabulary is not yet modeled.
- Authority precedence is implemented and tested, but unavailable-source fall-through for real providers is an open policy decision.
- Runtime EventStore/Dapr/Aspire evidence remains outside the current in-process workflow proof.

Recommended response:

- Do not start Epic 3 with implicit trust assumptions.
- Add capability/no-disclosure/policy decisions directly to Epic 3 story Dev Notes.
- Keep external confirmation scoped and evidence-based; do not let it become approval or a portal.
