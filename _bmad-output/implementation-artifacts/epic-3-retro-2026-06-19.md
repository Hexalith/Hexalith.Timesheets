# Epic 3 Retrospective: External Contributor Confirmation

## Epic Summary

Epic 3 delivered the external contributor confirmation surface for Timesheets.

- 5 of 5 stories completed.
- External API callers can submit or confirm external Time Entry evidence without becoming internal tenant users.
- External contribution commands reuse existing Time Entry record/submission services instead of creating a bypass path.
- Confirmation remains contributor evidence, not approval, locking, or ledger eligibility.
- Magic-link capabilities are modeled as opaque, scoped, single-use, expiring capabilities with token-hash persistence in domain events and no raw token exposure.
- Magic-link confirm and adjust flows validate scoped state before display/use and record EventStore-backed capability-use and Time Entry evidence.
- Invalid, expired, used, revoked, unauthorized, malformed, unknown, wrong-recipient, wrong-action, wrong-target, and repeated links collapse to the same external no-disclosure denial.
- FrontComposer metadata and OpenAPI schemas were extended without introducing a bespoke Timesheets UI project.
- Final story records report clean build validation and passing direct xUnit v3 lanes after local VSTest socket fallback.

Sprint status confirmed all Epic 3 story keys were `done`. The Epic 3 aggregate row still said `in-progress` before this retrospective, so this workflow corrected it to `done` along with marking `epic-3-retrospective` as `done`.

## Review Pattern Synthesis

The senior reviews found and fixed or tracked meaningful issues:

- Story 3.1: File List omission was fixed; an unreachable state-fabricating fallback was simplified; external contribution stayed on the existing record/submission path.
- Story 3.2: malformed scope handling was fixed so missing/unknown targets fail closed instead of throwing; workflow E2E test omissions were corrected.
- Story 3.3: a missing display path and untrusted audit-source channel were fixed; null host endpoint state remained a tracked follow-up.
- Story 3.4: adjustment validation and privacy held; `IMagicLinkConfirmationCapabilityStateLoader` was introduced, but only an unavailable fail-closed implementation was registered.
- Story 3.5: the overstated loader-replacement checkbox was corrected; outcome-vocabulary testing was strengthened; HTTP-boundary no-disclosure proof and telemetry recording remain tracked follow-ups.

The pattern is healthy after correction: no critical story-blocking defect remained when stories were marked `done`. The key remaining risk is not the domain behavior; it is live-host/persisted loader readiness and HTTP-boundary proof.

## Key Learnings

1. External confirmation must stay separate from approval in commands, events, projections, metadata, OpenAPI, and copy.
2. No-disclosure is a behavior matrix, not a single error message. It must cover status, payload, schema, route behavior, side effects, and diagnostics.
3. A fail-closed unavailable loader is safer than a fake persisted lookup, but it must be documented as not production-ready.
4. Service-level workflow tests are strong proof for domain rules; HTTP-boundary tests are still needed for external disclosure equivalence.
5. Token-hash and capability state are security-sensitive infrastructure. They need EventStore-backed load/fold design, not local caches, static dictionaries, projection mutation, or token inspection endpoints.
6. Metadata-only external UI descriptors are not a substitute for component accessibility tests once a UI package exists.
7. Story records need continued honesty about File Lists, test counts, and deferrals.

## Epic 4 Preview

Epic 4 is "Approved Time Ledger, Reporting & Finance Export." It will query time entries, project approved ledger evidence, produce actual-time reports, surface AI effort reporting, generate finance exports, verify export evidence, and add a dashboard overview.

Dependencies on Epic 3:

- Epic 4 query/report surfaces must include external contributor category and source type where filters require it.
- Approved ledger and export logic must treat external and magic-link evidence as contribution provenance, not approval authority.
- Magic-link confirmation and adjustment evidence should remain visible only through authorized read models and export policy.
- Finance/export paths must preserve no-token and no decoded-capability guarantees.
- Any Epic 4 dashboard or report shortcut that references confirmation state must not rely on the current unavailable default loader as live persisted truth.

Preparation needs:

- Decide whether Approved-Time Ledger/export rows include external source and magic-link confirmation/adjustment metadata beyond the current FR18 minimum.
- Keep finance export out of token, capability, invoice, payroll, rate, tax, and revenue-recognition ownership.
- Add report and ledger tests for external contributor category, source type, magic-link-adjusted entries, and no raw token/hash disclosure.
- Resolve or explicitly defer carry-existing-approved-entry behavior before ledger/export eligibility depends on it.
- Document the EventStore-backed magic-link state-loader requirement as launch-readiness work if live magic-link endpoints must function before finance workflows.

## Significant Discovery Assessment

No implementation learning requires rewriting Epic 4's core scope. Epic 4 remains directionally sound.

Significant but non-blocking discoveries:

- Magic-link live host behavior is fail-closed by default because the only registered state loader returns unavailable state.
- Invalid-link no-disclosure is strong at the service boundary, but HTTP-boundary equivalence tests are still needed.
- Outcome-category vocabulary exists but is not yet recorded in diagnostics/audit.
- External UI accessibility remains metadata-level until a Timesheets UI package exists.
- README and architecture notes needed updates to stay honest about Epic 3 coverage and current loader readiness.

Recommended response:

- Do not let Epic 4 treat confirmation as approval evidence.
- Do not let finance/export surfaces expose token, hash, capability, or reason-disclosure material.
- Decide whether the loader is launch-readiness work independent of ledger/report/export implementation.
- Keep report and export tests focused on external contributor category, source type, correction lineage, and privacy.

- Keep Epic 4 finance/reporting work grounded in approved evidence, not confirmation evidence.
