# Epic 1 Retrospective: Trusted Time Capture & Activity Governance

## Epic Summary

Epic 1 delivered the Timesheets foundation for trusted capture and activity governance:

- 9 of 9 stories completed.
- Time Entry capture works against exactly one Project or Work reference.
- Tenant and resource authorization gates fail closed before trusted work.
- Tenant and project Activity Type governance is modeled with stable IDs, active/inactive behavior, catalog restrictions, replay-safe projections, and FrontComposer metadata.
- Evidence retention and comment sensitivity policy vocabulary is in place, with fail-closed defaults for trust-bearing operations until policy is configured.
- Time Entry evidence read models expose source authority, event lineage, projection freshness, and read-time display hydration without copying sibling-owned data.
- AI-assisted Time Entry metrics preserve separate units for duration, wall-clock runtime, model/tool runtime, billable effort, source metadata, and token availability.
- Build validation passed with warnings as errors across the epic, and direct xUnit v3 test lanes were used where VSTest socket permissions blocked `dotnet test` in the sandbox.

Sprint status confirmed all Epic 1 story keys were `done` before this retrospective. No previous epic retrospective existed because this was Epic 1.

## Challenges

Amelia (Developer): "Several stories initially had claims that were directionally correct but not executable enough. Reviews found orphaned paths, source-scan-only tests, and missing File List entries."

Murat (Test Architect): "The recurring quality pattern was not failed builds. It was verification depth: tests sometimes proved a shape existed but not that the real composed service used it."

Winston (Architect): "The current implementation is still adapter-seam heavy. Tenant, Project, Work, Party, EventStore, and hydration integrations are intentionally fail-closed or unavailable by default. Epic 2 must not mistake those seams for production integrations."

Alice (Product Owner): "Evidence and comment policy is deliberately fail-closed. That is correct for trust, but submission and approval stories must make policy setup explicit or positive paths will be blocked by default."

Paige (Technical Writer): "README test instructions lagged behind implementation reality. Story records consistently used direct xUnit v3 executables in this sandbox, and Story 1.9 added an executable integration workflow, but the README still described integration tests as only reserved."

## Key Learnings

1. Review must verify executable wiring, not only declared abstractions.
2. File Lists are quality evidence. They should be checked against actual source changes before story completion.
3. `dotnet test` is still the canonical CI-style command, but this workspace needs a documented direct xUnit fallback because VSTest opens blocked sockets in the current sandbox.
4. Fail-closed defaults are working as designed, but they increase the need for explicit test fixtures and configured policy/adapters in the next epic.
5. Projection behavior needs behavioral tests for replay ordering, idempotency, freshness, and disclosure, not only source scanning.
6. AI metrics must remain multi-unit evidence. The implementation correctly avoids converting token counts, runtime, or wall-clock values into human duration.
7. FrontComposer metadata is sufficient for Epic 1. A dedicated Timesheets UI project is still intentionally deferred.

## Review Pattern Synthesis

Across Epic 1, adversarial reviews repeatedly found medium-severity gaps that were fixed before approval:

- Story 1.1: host/server/service defaults were not wired; telemetry was not configured; Projections tests were initially source-scan only.
- Story 1.2: UI action authorization outcomes were declared but not produced; positive dispatch and composed fail-closed defaults needed tests.
- Story 1.3: duplicate lifecycle vocabulary was removed; metadata badge coverage was added.
- Story 1.4: dead policy evaluator code was removed; comment length bounds were added.
- Story 1.5: unused rejection vocabulary and dead state method were removed; replay ordering was made deterministic.
- Story 1.6: missing project-reference and tenant-before-project authorization paths were covered; label collision and out-of-order replay tests were added.
- Story 1.7: a misleading invalid-comment test was replaced with genuine comment-policy validation.
- Story 1.8: File List omissions and missing privacy fitness coverage were corrected.
- Story 1.9: integration test files were added to the File List, and unavailable AI metric validation was made unconditional.

The pattern is positive after correction: no critical findings remained when stories were marked `done`, and the final story reports full build/test success.

## Epic 2 Preview

Epic 2 is "Submission, Approval, Period Review & Corrections." It depends on Epic 1's Time Entry capture, Activity Type catalogs, fail-closed authorization, policy evaluation, and evidence projection.

Expected dependency points:

- Submission must validate captured Time Entries through the existing Time Entry model, Activity Type catalog freshness, target references, contributor references, and comment policy.
- Approval authority must reuse tenant/resource gates and add approver-specific policy without caller-supplied authority.
- Approved-entry locking and correction must be additive domain facts, not direct mutation of recorded evidence.
- Period review must preserve entry-level state and should not flatten mixed entry states.
- Read models and queues must expose projection freshness and non-disclosing authorization outcomes.

Readiness assessment:

- Story-level completion: complete.
- Quality evidence: build and direct xUnit lanes green in story records.
- Deployment evidence: no production deployment evidence was present in Epic 1 records.
- Stakeholder acceptance evidence: no external stakeholder acceptance artifact was present; acceptance is represented by story review approval.
- Technical health: suitable for Epic 2 foundation work, with explicit integration seams still requiring configured adapters/fixtures.
- Unresolved blocker: no blocking code defect was identified, but Epic 2 must treat policy/adapters/EventStore integration as real dependency work, not assumptions.

Significant discovery assessment:

- No implementation learning requires rewriting Epic 2's scope.
- The current Epic 2 plan is still directionally sound.
- Preparation should focus on adapter/policy fixture clarity, approval authority semantics, and correction lineage tests before positive approval workflows are considered complete.
