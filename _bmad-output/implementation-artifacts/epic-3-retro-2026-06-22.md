# Epic 3 Retrospective (Closure): External Contributor Confirmation

## Why This Retrospective Exists

The first Epic 3 retrospective (2026-06-19) reviewed 5 stories and flagged two headline gaps as its top action items:

1. The registered `IMagicLinkConfirmationCapabilityStateLoader` was the fail-closed `Unavailable` stub — no concrete EventStore-backed loader existed.
2. No-disclosure was proven only at the service/workflow layer — there was no executable proof at the HTTP boundary external contributors actually hit.

The 2026-06-20 readiness repair turned those two gaps into owned feature work — Story 3.6 (EventStore-backed loader) and Story 3.7 (HTTP-boundary no-disclosure proof) — and reopened Epic 3 to `in-progress`. Both stories are now `done`. This retrospective closes Epic 3 across all 7 stories and verifies the prior commitments were honored.

## Epic Summary

Epic 3 delivered the external contributor confirmation surface for Timesheets: API-only external submission/confirmation, scoped opaque single-use magic links, magic-link confirm and adjust flows, uniform no-disclosure rejection of invalid links, a concrete EventStore-backed state loader, and executable HTTP-boundary proof of no-disclosure.

- 7 of 7 stories completed (3.1–3.7).
- External API callers submit or confirm Time Entry evidence without becoming internal tenant users, reusing the existing record/submission services rather than a bypass path.
- Confirmation remains contributor evidence — never approval, locking, or ledger eligibility.
- Magic-link capabilities are opaque, scoped (tenant, contributor Party, target entry, allowed action, expiry, single-use), persisted as token hashes in domain events, with no raw token exposure.
- Invalid links — malformed, unknown, expired, used, revoked, unauthorized, cross-tenant, wrong-recipient, wrong-action, wrong-target, stale-catalog, and replayed — all collapse to the same opaque external denial.
- Story 3.6 added the concrete `EventStoreMagicLinkConfirmationCapabilityStateLoader` (registered, replacing the `Unavailable` default), a new rebuildable non-authoritative token-hash candidate index, and a new `ITimesheetsTrustedContextAccessor` tenant-context seam.
- Story 3.7 added the module's first over-the-wire `WebApplicationFactory<Program>` tests proving equivalent opaque 403 responses across 11 invalid cases × 4 external routes, plus correlation-safe denial diagnostics (`LogExternalLinkDenial`, EventId 37001).
- FrontComposer metadata and OpenAPI schemas were extended without introducing a bespoke Timesheets UI project; the invalid-link outcome vocabulary stays server-internal and off the external contract.

**Sprint-status correction:** all seven Epic 3 story keys are `done`, but the `epic-3` aggregate row still read `in-progress` (the 2026-06-20 repair reopened it to add 3.6/3.7). This retrospective corrects `epic-3` to `done` and marks `epic-3-retrospective` as `done`.

- 3.5: the overstated loader-replacement checkbox was corrected to a tracked deferral; outcome vocabulary strengthened; HTTP-boundary proof and telemetry recording left as tracked follow-ups (→ 3.6/3.7).
- 3.6: catalog freshness made deterministic (clock-free `AsOfUtc`); a dead DI registration removed; token-hash index built as a non-authoritative candidate resolver with the authority split guarded by test names.
- 3.7: accidental `Hexalith.Tenants` submodule bump reverted; inaccurate test counts corrected; production endpoints changed minimally (denial diagnostics) while keeping the single-`Denied()`/single-`Results.Problem(` fitness invariant green.

The persistent theme is honesty discipline — File Lists, test counts, and `[x]` marks — caught reliably by adversarial review, not by self-reporting.

## Key Learnings

1. **Story-complete is not launch-ready, and a checkbox is not proof.** Story 3.7 exists because Story 3.5's checkbox overstated completion. Behavior must be proven to execute at the boundary external callers hit.
2. **No-disclosure is a behavior matrix, not one error message.** Status, content type, ProblemDetails body, headers, side effects, and diagnostics must all be equivalent across every invalid category — proven over the wire, with a non-vacuous success case so the assertion can fail.
3. **Token→capability resolution needs a dedicated, non-authoritative index.** A candidate resolver folded from `Issued` events is fine for lookup, but single-use/revoked/expired truth must come from folding the authoritative aggregate. Do not blur the two.
4. **Determinism must not depend on the system clock.** The sequence cursor already conveys freshness; reaching for `DateTimeOffset.UtcNow` in a fold breaks idempotency.
5. **Fail-closed defaults plus host replacement is the right composition shape.** The kernel keeps `Unavailable`/`DenyAll` defaults; the host replaces them (`HttpContextTimesheetsTrustedContextAccessor`). Test doubles belong in the test host, never in production DI.
6. **A registered concrete loader is necessary but not sufficient for live operation.** The token-hash candidate index still needs live projection-host wiring before a valid magic link resolves end-to-end. This is honest launch-readiness work, not a hidden defect.
7. **Metadata-only external UI descriptors are not accessibility tests.** NFR13 remains deferred until a Timesheets UI package exists; do not scaffold UI just to claim coverage.
8. **Adversarial review is load-bearing.** Every recurring honesty lapse (File Lists, counts, `[x]` marks, submodule drift) was caught at review, not at authoring.

## Significant Discovery Assessment

No implementation learning requires rewriting Epic 4's core scope. Epic 4 remains directionally sound.

Significant but non-blocking discoveries:

- The concrete EventStore-backed loader exists and is registered, but the token-hash candidate index has no live projection-host wiring, so a valid magic link does not yet resolve end-to-end in the running topology. Invalid-link no-disclosure is fully proven regardless.
- A new `ITimesheetsTrustedContextAccessor` seam was introduced during implementation (not in the frozen Story 3.4 interface design) to supply tenant context to the tenant-less loader interface.
- The token-hash index is a non-authoritative candidate resolver; validity authority remains the folded capability aggregate.
- `ConfirmAsync` does not consume the Activity Type catalog, so the projection-freshness gate applies only to the adjust/describe paths — a deliberate behavioral asymmetry.
- NFR13 external-UI accessibility remains metadata-level until a Timesheets UI package exists.

Recommended response:

- Treat live magic-link resolution (projection-host wiring) as explicit Epic 5 launch-readiness work, independent of ledger/report/export implementation.
- Do not let Epic 4 treat confirmation/adjustment as approval evidence.
- Keep finance/export surfaces free of token, hash, capability, and reason-disclosure material.

## Commitments

- Keep external confirmation, adjustment, approval, correction, and ledger eligibility as separate concepts.
- Keep magic-link no-disclosure behavior equivalent across all invalid states, proven at the HTTP boundary.
- Keep token values, decoded capability material, comments, personal data, target names, and protected identifiers out of logs, OpenAPI invalid responses, exports, and diagnostics.
- Keep the token-hash index non-authoritative; validity authority stays with the folded capability aggregate.
- Keep fail-closed kernel defaults until the host or a real loader explicitly enables a path.
- Keep Epic 4 finance/reporting work grounded in approved evidence, not confirmation evidence.
